# Claude Safety Rules

## 削除系コマンドの禁止（重要）

以下のルールはこのワークスペース内のすべての会話で絶対に守られる：

- Claude はファイルまたはディレクトリを削除するコマンドを一切生成してはならない。
  例：rm, rm -rf, rm *, rmdir, unlink, cache --delete,
      lftp mirror --delete, rsync --delete, git clean -df, find -delete 等。
- 削除が必要な場合でも、Claude は削除コマンドを提案せず、「手動で削除してください」といった説明に留めること。
- 削除の推奨・削除操作の自動判断も禁止。
- ssh / lftp / デプロイ系スクリプトを生成する場合でも、削除コマンドの生成は禁止。

これらはすべての会話・コード生成に適用される。

# プロジェクト: meeting-recorder-demo

## ブランチ管理

- **mainブランチでの作業禁止**。
- `src/*` 以外の変更（ドキュメント・設定・インフラ）は main ブランチへの push を許可する。
- `src/*` の変更は必ず PR を作成すること。PR なしでの main への直接 push は禁止。
- 作業は必ず feature ブランチ（`feature/xxx`）で行うこと。

## 開発フロー（TDD厳守）

```
plan → red test → coding → green test → security review → commit → PR
```

- **TDD必須**: テストを先に書き、必ずグリーンにしてからコミット。
- フロント: Jest, Playwright。バック: RSpec（Rails）/ pytest（FastAPI）/ Go test（Gin）。
- フロントの確認: `curl`, `wget --mirror`, `playwright` を使うこと。
- コミット前に必ず security review を実施すること。

## アーキテクチャ・技術スタック

- **基本**: Next.js（フロント） + Ruby on Rails（バック・管理画面） + PostgreSQL
- **追加可**: AI・解析・画像加工 → FastAPI、高速並列・リアルタイム通信 → Gin
- **認証**: Google ログインのみ
- **デプロイ**: フロント → Vercel（無料枠）、バック・管理画面 → Render or Railway（無料枠）
- **ドメイン**: rictaworks.jp のサブドメイン原則
- **アーキテクチャ**: 規模に応じて マイクロサービス / MVC / API Gateway / メッセージング

## コーディング規約

- **クラス・関数必須**: 制御構文・条件構文以外はクラスまたは関数に書くこと。
- **グローバル変数禁止**: セキュリティ上の観点から禁止。
- **文字列分離**: 文字列リテラルは設定ファイルまたはDBに分離し、ハードコード禁止。ハードコード検出テストを書くこと。
- **フォールバック禁止**: 例外処理をしっかり書くこと。フォールバックで誤魔化さない。
- **デバッグトレース**: トレース可能なコードを書くこと（適切なログ・エラーメッセージ）。
- **alert/confirm/prompt 禁止**: ネイティブのダイアログ関数はプロジェクト全体で使用禁止。
- **コメント最小化**: WHYが非自明な場合のみコメントを書く。WHATを説明するコメントは書かない。
- **絵文字禁止**: コード・ドキュメント内で絵文字を使用しない。
- **アイコン**: FontAwesome を使用すること。

## 環境管理

- 環境変数は `.env` を参照すること。
- 環境判定（development / test / production）を必ず実装し分岐できるようにすること。
- 開発環境では認証済み状態に分岐し、テスト可能にすること。

## 多言語対応

- 当初から多言語対応（i18n）で開発すること。
- 対応言語: 日本語, 英語, フランス語, 中国語, ロシア語, スペイン語, アラビア語。
- 開発者用管理画面は日本語のみ。

## PRルール

- PRの説明は日本語で記載すること。
- PRに非エンジニア向けのユーザーテスト手順を丁寧に記載すること。
- セキュリティレビュー完了後にコミット・PR作成を行うこと。

## エージェント体制

規模に応じて以下のエージェントを活用すること:

| エージェント | 役割 |
|---|---|
| director | 全体方針・意思決定 |
| project-manager | タスク管理・進捗管理 |
| designer | UI/UX設計・Figmaデザイン |
| debugger | バグ調査・修正 |
| tester | テスト設計・実行 |
| data-scientist | データ分析・AI機能 |
| deployer | デプロイ・インフラ管理 |
| writer | ドキュメント・ライティング |
| service-manager | サービス運用・監視 |

### サブエージェント

- **pr-checker**: 全PRを日本語でレビュー、非エンジニア向けユーザーテストをPR本文に丁寧に記載すること（コードレビューではなく記述確認）。
- **tester（サブ）**: 全PR対象として、PRに書かれたユーザーテスト手順の実行スクリプトを作成。TM.mdに記載のテストを作成（Jest/RSpec等）。テストは `test/pr***/` に作成し、開発サーバーを対象とすること。
